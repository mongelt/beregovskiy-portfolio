## Definitions
**Resume page / Resume tab**: the tab of the front page that this document describes
**Timeline**: The dynamic, flexible central green line that runs from the present to the past and has resume entry card on it and its; consists of month markers that adjust their height to fit resume entry cards
**Green or central line**: The vertical green line on the page is referred to as the "green line" or "central line"
**Resume entries**: Entries created in the admin panel that have a title, start and end dates, and additional information 
**Resume entry cards**: Resume entry cards are a a display of resume entries on the Resume page
**Month marker**: The building block of the central timeline that constructs the timeline and adjusts its height to fit resume entry cards
**Standard card**: The entry card for longest resume entry out of the entire list of entries
**Standard month marker height**: The height of one month marker for the standard card
**Start and Now markers**: Month markers at the beginning and the end of the timeline, all other month markers fit between Start and Now markers
**Activated month markers**: Month markers become activated when there's a resume entry with a start or end date that matches the month marker; activated month markers are displayed on the timeline (except for the missing end date)
**Operational month markers**: Month markers between Now and Start markers are always present (unless there are no resume entries) but are not displayed until they are activated; operational markers expand and shrink depending on activated markers, and are displayed (but don't affect the timeline logic) in the marker debugging mode
**Side lines**: Each resume entry card has a side line that runs alongside the green central line only for the duration of the associated resume entry card.
**Main debug window**: Displays the box on top on the time (above the Now marker) and shows usefull information for debugging
**Marker debug window**: Displays all operational markers but doesn't activate them; displayed operational markers don't change the timeline logic, flexibility, or height
**Featured entries and featured entry cards**: Entries are featured by a checkbox control in the admin panel; featured entries show on the timeline, non-featured entries are stored in the backend and admin, but are not displayed on the page and timeline

## Timeline Core Logic
### Timeline basics
**Resume entries**: The timeline consists of resume entries pulled from the admin panel
**Entry cards**: Entry cards are boxes of elements that are displayed on the page
**Central timeline**: At the center of the screen runs a green vertical line from the past to the present
**Month markers**: Resume entries have start and end dates, which create month markers
**Display Order**: Resume entry cards are displayed on the timeline from the newest entries on the top to to the oldest entries on the bottom on the page
**Side and center entries**: Entry cards are displayed on left or right sides of the timeline (side cards), or at the center of the timeline (center cards)
**End date markers**: End date markers define the order: an entry card that has a later end date is placed on top of an entry with an earlier date (end date of December 2024 is lower than end date of January 2025); the entry that ended LAST should be positioned FIRST (higher on timeline); if end dates match, see the Conflicts section
**No end date**: The "Present" mark is not displayed, it is replaced by the Now mark
**Missing end date**: If the end date is missing on side entries, treat as "Present" and position at current date at the Now marker
**Missing start date**: if a side entry is missing a start date, treat the start date as equal to the end date for marker purposes and show only the end month marker; in the date range display on the card, show only the end date (not "Present → End Date")
**Date normalization**: All dates used for timeline calculations must be normalized to the first day of the month in Eastern Standard Time to ensure consistent month counting across timezones. The current month for the Now marker is computed in Eastern Standard Time.

## Timeline main elements and behavior 
### Basics
**Meaning behind central timeline**: The green timeline represents years running back: it flows from OLDEST (bottom of the page) to NEWEST (top of the page) 
**Start of the line**: The line (the past) starts at the bottom of the page at the Start marker, which is the earliest start date of all entries
**End of the line**: The line (the present) ends at the top of the page at the Now marker, which is current month
**Month markers**: Month markers go back from the Now marker at the top to the Start marker at the bottom, the top month marker on the page is the previous month relative to now
**Activated Month Markers**: Month markers are activated by start and end dated of resume entries and are displayed for start and end dates of each entry. 
**Operating markers**: All month markers between Start and Now markers are always operating and participate in timeline behavior, even if they're not displayed
**Activated month markers logic**: If an entry card is too long to fit between activated start and end month markers, activated markers and all operating markers in between activated markers increase in height as much as needs to expand the timeline to fit the entry card
**After the end of the timeline**: After its end at the earlier start date of all entries, the green line continues for 300px fading away and ends with a caption "Born in Moscow, Russia - July 1st, 1994" placed at the center
**Side lines**: Side lines are created by resume entry card and they run between month markers alongside the green central line on the same side as the entry card for the entire duration of that entry card; they have a different color than green 

## Month markers
**Activation**: Month markers are activated and displayed by resume entries start and end dates
**Center and side entries**: Side entries activate green markers, center entries activate light blue markers; green: emerald-400; light blue: `#88b6e3`

## Month markers and timeline dynamics
**End date markers for side entries**: Side entries cards are aligned with month markers according to their end dates
**Start date markers for side entries**: The end for side entries cards cannot be lower than month markers according to their start dates
**Center entries**: Center entries are on the timeline in its center and in the middle between start and end date markers
**Dynamically flexible timeline**: Start and date month markers move on the page to fit the resume entry card; if the resume card is too long to fit between the end month marker on the top and the start month marker on the bottom, they need to move on the page as much as needed to fit the entry card
**Collapsed/expanded resume entry card**: Entry cards collapse and expand; month markers need to move accordingly with collapsed and expanded states of entry cards
**Month marker height**: Month markers have a height, which is how many pixels does a month marker needs to expand the timeline to fit resume cards

## Center Entry Special Logic

### Month Marker Colors for Center Entries
- Center entry month markers use RGB `#88b6e3` instead of emerald-400
- Both start and end date markers for center entries use RGB 88b6e3

### Center Entry Expansion
- Center entries have an Expand button that is shown on the structure scheme as [▼] and looks like an arrow going down when collapsed and going up when expanded
- Expands to show `short_description` plain text
- Uses line-based height estimation: 24px per line (1-2 lines max)
- Calculation: text length ≤60 characters = 1 line (24px), >60 characters = 2 lines (48px)
- Simple text content uses estimation rather than DOM measurement
- Arrow only appears if short_description exists
- Background height adjusts based on arrow presence
- Timeline adjusts to expansion of the entry

### Center Entry Markers
- Center entries have blue date markers within their cards (start/end dates)
- These blue markers replace green markers on the main timeline (regular green markers are never displayed for center entries, only the blue)
- Blue markers are counted separately in the debug window using `centerEntryDates` computed with `useMemo`
- If a center entry doesn't have a start date, override the default logic; position the center entry as usual at the end date, don't display the start month marker, only display the end month marker
- If a center entry doesn't have an end date, treat the start date as the end date FOR ALL TIMELINE PURPOSES: (1) Month counting: count as 1 month (start_date == end_date), (2) Entry sorting: use start_date as end_date for ordering calculations, (3) Timeline positioning: position at the start_date location (treated as end_date), (4) Display: show only one date on card (as if start_date was missing). This ensures center entries without end dates are positioned chronologically by their start_date, not at "Present" like side entries.

## Dynamic Timeline

### Timeline boundaries
**Start of the dynamic timeline**: The dynamic timeline starts at the bottom of the page at the Start marker and represents the past
**End of the dynamic timeline**: The dynamic timeline ends at the top of the page at the Now marker and represents the present

### Marker System
- **Now Marker**: Always active, always at top, displays as "Now", current month
- **Start Marker**: Scans all entries and identifies the earlist start date from all entries
- **Month Markers**: First day of each month (e.g., "October 2025" = October 1, 2025 EST)
- **Birth Caption**: Caption that says "Born in Moscow, Russia - July 1st, 1994" is always visible, located 300px below the Start marker but doesn't take any part in dynamic timeline

## Dynamic timeline behavior
### Standard card and standard month marker height
**Resume entries activated**: When there's at least one marker activated, the timeline starts dynamics in the following order:
1. The timeline scans for the resume entry card that takes the longest period of time between start and end dates and defines it as a "standard card"
2. Counts the number of months in the standard card including start and end months
3. Measures the height of standard card in collapsed state in pixels
4. Divides the height of the standard card by the number of months and rounds it to the nearest integer
5. Identifies that as the month marker height
**Standard month marker height calculation**: The result of the division of the height of the card for the resume entry that takes the longest period of time between start and end dates by the number of months this entry takes is the standard month marker height
**Only featured entries**: The standard card is chosen only from the entries that are featured (for a difference between featured and non-featured entries, see the Definitions section)
**Featured entries scope**: All timeline calculations consider only featured entries, including standard card selection, month marker activation, Start marker calculation, and timeline expansion. Non-featured entries are excluded from all timeline logic.
**Two side entries have the same duration/tie-breaking**: If multiple entries tie for the longest period (same number of months), choose the entry with the lowest end date; if still tied, choose the one with the latest start date; if still tied, choose the topmost by display order.
**Expansion of standard card**: Standard month marker height is measured only based on the collapsed state of the standard card and doesn't change when the standard card is expanded. 
- *NOTE*: the standard card can expand, and when it is expanded, the timeline adjusts between start and end month markers just like with any other entry; but the standard month marker height stays the same and doesn't change. 

### Dynamic expansion
**Empty state**: If the timeline is empty and there are no entries, then the system cannnot identify a Start marker, and only displays the Now marker and the Birth Caption with a 300px line, no other markers are active
**Month markers are always operating**: If there is at least one entry, all month markers between the start and the end of the timeline are always operating and have a height, no matter if they are activated by entries to be displayed or not
**Filling the timeline**: After identifying the standard card and the standard month marker height, the timeline gets filled by operational month markers between the start of the timeline and its end, and they all have standard height
- *Single entry*: If there's only one entry activated, the timeline stays the way it is described in **Filling the timeline**
**Moving month markers**: If there is more than one entry and there's an entry card that is too long to fit between start and end month markers, the timeline adjusts itself to fit the entry card by expanding the height of all month markers between start and end date markers
**Month markers expanded from standard height**: If the timeline needs to expand to fit an entry card, month markers between start and date (including start and end date markers) expand proportionally to fit the entry card
**Start and end month markers expansion**: Markers between start and end months expand outward from their own center, with equal expansion above and below the marker's center; start month markers expand up (toward the Now marker); end month markers expand down (toward the Start marker); see Example 3 illustrating this logic as an example
**Start and end dates match**: If start and end dates for a side entry match, or the start date is missing and treated as the end date, the activated month marker (that acts as both start and end month marker for this entry) expands like an operational marker would usually expand outward from the center of the marker, both up and down equally
**Side entry lasts for two months**: If a side entry lasts for only two months, it activated two month markers that expand like start and end month markers usually would; no operational markers in between are activated. 
**Expanded month markers height**: The expanded month marker height is defined as the height of the entry card in pixels divided by the number of month markers in that entry (start and end month markers plus every operational marker between start and end), and rounded to the nearest integer
**Rounding precision**: When rounding to the nearest integer results in exactly .5, apply tie-breaker rules: start markers round up, end markers round down, operational markers split the value equally (round the upward half up, the downward half down).
**Expanded state**: If the entry is expanded and doesn't fit in expanded state, month markers move to fit the expanded state; no matter if the entry is collapsed or expanded, the timeline adjusts to it
**Gaps**: Month markers in gaps between entries have standard height
**Overlapping entries**: If there are entries with overlapping dates, and both of them need the timeline to expand to fit, the height per affected month marker is the maximum required height among all overlapping entries for that month.
**Center entries**: Center entries have fixed predetermined heights (4 possible sizes: collapsed with/without arrow, expanded with 1/2 lines per lines 220-224) unlike side entries with variable heights. Despite fixed heights, center entries participate fully in timeline dynamics - their blue month markers expand/contract according to standard expansion rules (lines 281-287) to fit their fixed card heights. Blue markers (#88b6e3) function identically to green markers (emerald-400) except for color. Center entries can never be selected as the standard card (line 270 refers to featured side entries only)
**Entry side lines**: Side lines for resune entry cards are dependent in the height and expansion of resume entry cards, month markers, and the central timelime

## Conflicts
**Same end dates for entries**: If end dates are the same for two entries, the one with the later start date goes first in the order and the one with an earlier start date goes second; this applies only to featured entries
**Two entries with overlapping dates on the same side**: If there are two or more entry cards that are on the same side of the timeline, the cards follow the standard order: latest end date goes first, earliest end date goes last, then latest start date goes first, earliest start date goes last

## Side lines
**Entries have side lines**: Every side entry (left or right) has a side line that runs along the central green line
**No side lines for center entries**: Center entries are in the middle of the green timeline and don't have side lines
**Colors for side lines**: Side lines colors complement the main green color of the timeline, and the first 18 side entries have their own individual side line color; only side entries after the 18th reuse the same colors; the list of side lines colors is available in the Styling Requirements section of this document

- September 2015, October 2015, November 2015 markers are operational and exist between activated markers (August 2015 and December 2015), and even though they are not displayed on the timeline, they expand as well to help fit Entry 2 card; since they are operational, they expand in both directions (up toward Now and down toward Start) equally to get to required 60px, which means they all need to add 54px in height, and to achieve that, they expand 27px in each direction up and down

### Example 4: Test Entries Behavior
- **Example** The current month is October 2025. The entries and heights of their cards are:
- Entry 1: April 2025 to present (right, 400 px)
- Entry 2: December 2021 to January 2025 (left, 200 px) 
- Entry 3: May 2019 to July 2019 (center) 
- Entry 4: September 2017 to December 2017 (right, 600 px)
- Entry 5: September 2010 to June 2014 (right, 400 px)

The timeline then behaves the following way:
1. Finds the longest entry (Entry 5) and defines it as standard, calculates the number of months including start and end months (46), finds the height of the standard card in pixels (400), divides the height of the standard card by number of months and rounds it to 9 px. Defines standard month marker height as 9 pixels. 
2. Sets every month marker between September 2010 and October 2025 at 9px and starts placing entries on the timelime
3. Calculates the number of months in Entry 1 as 7, divides 400px by 7, rounds to the nearest integer, gets 57, compares 57 to standard height, understand 57 is larger than 9, sets 57px per each operational month marker to fit Entry 1 card
4. Calculates the number of months in Entry 2 as 38, divides 200px by 38, rounds to the nearest integer, gets 5, compares 5 to 9, understands 5 is smaller than 9, leaves month markers at standard height
5. Identifies Entry 3 as center, calculates the number of months as 3, sets the entry height at 192px, divides 192 by 3, gets 64, sets the height for May 2019, June 2019, July 2019 markers at 64px.
6. Calculates the number of months in Entry 4 as 4, divides 600px by 4, gets 150, sets the height for markers September 2017 to December 2017 inclusive as 150
7. Leaves other markers at 9px and renders the timeline

## Debugging Modes
**Two debug modes**: This development uses two debug windows: the main debug window and the marker debug window
**Activation**: Both debugging windows are activated in the admin panel.
**No effect on the logic**: Both debug modes exist only to provide information and a look into current development, they have no effect on the timeline logic or placement (other than the main debug window moving the timeline down to fit the window)

### Main debug mode
**Purpose**: Get an insight into the real behavior of the page, loaded entities, and expansion of the timeline
**Placement**: If activated in the admin, the main debug window is placed above the timeline and it moves the timeline down
**Information displayed**: The main debug window displays the following information:
1. Number of loaded entries that are featured on the timeline (to understand which entries are featured, see the Definitions part of this document)
2. Number of expanded entries
3. For each loaded entry: location (left/right/center), state (collapsed/expanded), end month marker, start month marker, entry card height in collapsed state, entry card height in expanded state
4. Number of activated month markers
5. Number of operational month markers
6. Total height of the timeline
7. Standard height in pixels, Title, start and end months of the entry card selected as standard card
8. Show markers (expand/collapse on click): when expanded, shows the label of each displayed marker (activated only if the marker debug is off; activated and operational if the marker debug is on)

(2) Render entry cards in collapsed state with `visibility: hidden` or `opacity: 0` initially. (3) Measure all collapsed heights using `getBoundingClientRect` in a single batch. (4) Calculate timeline layout based on measured heights. (5) Apply calculated positions and make cards visible. (6) Cache measured heights by entry ID and only re-measure when entry content changes. For expanded states, repeat steps 2-5 only after user expands an entry. This prevents timeline jumping or flickering during initial load.

- Overlapping expansion algorithm (Approach A - Calculate-All-Then-Apply-Max): For Steps 4.6-4.7, use two-pass approach per line 290. Pass 1: Loop through all entries, calculate required height per month for each entry (card height ÷ month count), store in Map<monthKey, Map<entryId, requiredHeight>>. Pass 2: Loop through all months in timeline, for each month look up all entries spanning that month, take Math.max() of all required heights, apply maximum as month marker height. This approach clearly implements 'maximum required height among all overlapping entries for that month' and makes debugging easier (can inspect all requirements before applying). Alternative sequential processing would work but makes it harder to verify correctness.

- Debug window data consistency: Debug windows should read directly from the same memoized computed state that drives rendering. Don't recompute values separately for debug display. This ensures debug numbers always match what's actually rendered.

- Resize and font scaling handling: Add resize event listener with debounce (200ms) to re-measure all cards and recalculate timeline. Only trigger recalculation if any card height changes by more than 1px to avoid unnecessary thrashing. Handle font size changes via CSS custom properties or media queries rather than direct measurements.
